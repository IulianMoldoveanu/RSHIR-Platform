# Native Shell — HIR Curier

**Status (2026-05-04):** infrastructure staged, **not yet shipped**. The
courier app runs as a PWA today (`courier-beta-seven.vercel.app`). This
doc explains when, why, and how we wrap it in a native iOS + Android
shell, and what gaps remain.

## When to ship native

**Trigger:** ~50+ active couriers OR first complaint that the PWA
"keeps logging me out" / "doesn't ring when phone is locked" /
"can't add to home screen on this phone" from a real rider.

Until that signal hits, the PWA is good enough:

- Web Push works on Android Chrome + iOS 16.4+ (with home-screen install).
- Geolocation + camera work in-browser.
- Service worker handles offline active-order recovery + offline fallback.
- Updates ship instantly with every `vercel deploy` — no app store review.

## Why Capacitor (not React Native, not Expo, not Tauri)

| Option       | Pro                                | Con                                              |
| ------------ | ---------------------------------- | ------------------------------------------------ |
| **Capacitor** | Reuse 100% of the existing Next.js app. WebView + native plugins. ~1 day setup. | Slightly heavier than RN. App-store review still required for native plugin updates. |
| React Native | Native UI feel.                   | Full rewrite of every screen. ~4-6 weeks.        |
| Expo          | RN + tooling.                      | Same rewrite cost. EAS Build is great but doesn't help if there's no RN code. |
| Tauri Mobile | Rust backend, small binary.        | Mobile beta as of 2026-05. WebView2/WKWebView coverage uneven. |
| **Decision** | Capacitor. The app is already a PWA; wrapping is additive. |

## Activation runbook (when Iulian green-lights)

```bash
cd apps/restaurant-courier

# 1. Install Capacitor + needed plugins.
pnpm add @capacitor/core
pnpm add -D @capacitor/cli
pnpm exec cap init "HIR Curier" "ro.hir.courier" --web-dir out

# 2. Add platforms.
pnpm add @capacitor/ios @capacitor/android
pnpm add @capacitor/geolocation @capacitor/push-notifications \
         @capacitor/splash-screen @capacitor/status-bar
pnpm exec cap add ios
pnpm exec cap add android

# 3. Sync after every web change.
pnpm build && pnpm exec cap sync

# 4. Open native projects.
pnpm exec cap open ios       # opens Xcode (macOS only)
pnpm exec cap open android   # opens Android Studio
```

`capacitor.config.ts` is already committed — it points the WebView at
the production Vercel deployment, so web releases ship to the native
shell instantly without an app-store resubmit. Only changes to native
plugins (geolocation, push, version bump) require a native rebuild.

## Permission strings — REQUIRED for review

These strings are **mandatory** for App Store + Play Store approval.
They live in native config (`Info.plist` for iOS,
`AndroidManifest.xml` for Android), generated by `cap add`. Drop these
exact strings in when prompted (RO + EN both — Apple requires native
language and English):

### iOS — `ios/App/App/Info.plist`

```xml
<key>NSLocationWhenInUseUsageDescription</key>
<string>HIR Curier folosește locația pentru a vă arăta comenzile din apropiere și pentru a calcula distanța până la restaurant și client.</string>

<key>NSLocationAlwaysAndWhenInUseUsageDescription</key>
<string>HIR Curier urmărește locația în fundal în timpul unei livrări active pentru a actualiza ETA-ul către client. Tracking-ul se oprește automat când livrarea se încheie.</string>

<key>NSCameraUsageDescription</key>
<string>HIR Curier folosește camera pentru dovada livrării (poză colet la predare).</string>

<key>NSPhotoLibraryAddUsageDescription</key>
<string>HIR Curier salvează dovada livrării în galerie ca backup local.</string>
```

### Android — `android/app/src/main/AndroidManifest.xml`

```xml
<uses-permission android:name="android.permission.INTERNET" />
<uses-permission android:name="android.permission.ACCESS_NETWORK_STATE" />
<uses-permission android:name="android.permission.ACCESS_FINE_LOCATION" />
<uses-permission android:name="android.permission.ACCESS_COARSE_LOCATION" />
<uses-permission android:name="android.permission.ACCESS_BACKGROUND_LOCATION" />
<uses-permission android:name="android.permission.CAMERA" />
<uses-permission android:name="android.permission.POST_NOTIFICATIONS" />
```

Play Store will additionally require an **in-app rationale** for
background location — show a one-time bottom sheet on first shift
start explaining why. This is built into `dashboard/shift/page.tsx`
already; verify before submission.

## Push notification keys — current state vs needed

| Channel                   | State | Notes |
| ------------------------- | ----- | ----- |
| Web Push (VAPID)          | Live | `vapid_public_key` + `vapid_private_key` in vault. Used by Edge Function `dispatch-push`. |
| **FCM (Android native)**  | **MISSING** | Need to create a Firebase project (or Google Cloud Messaging legacy), download `google-services.json`, drop in `android/app/`. Then update `dispatch-push` to send FCM payloads when target is native (vs Web Push). |
| **APNs (iOS native)**     | **MISSING** | Need an APNs Auth Key (`.p8`) from the Apple Developer account, configure in Capacitor + `dispatch-push`. APNs supports both VoIP and standard push; standard is enough for our use case. |

When Iulian provisions Apple Developer + Firebase: store the `.p8`
key + `google-services.json` in the secrets vault under
`apns.auth_key_p8` and `firebase.android_config_json`. The Edge
Function `dispatch-push` can then route by target platform (web /
ios / android) on the `push_subscriptions` row.

## Splash screen assets

`public/splash.html` is committed as a fallback HTML splash (used by
some shell configurations). For Capacitor native splash you need:

- **iOS:** `splash@1x.png` (320×480), `@2x` (640×960), `@3x` (1242×2208).
  Or, easier: a single 2732×2732 universal PNG and let `cordova-res` /
  `cap-res` slice it.
- **Android:** `splash.png` 2732×2732 + adaptive icon foreground/background
  layers. Drop into `android/app/src/main/res/drawable*` after `cap sync`.

## Manifest icon coverage

Currently shipped: 192×192 + 512×512 PNGs (`/icon-192.png`,
`/icon-512.png`). Both declared `any` and `maskable` in the manifest.

**Gaps a designer should close before app-store submission:**

- 1024×1024 master icon (App Store + Play Store listing) — non-PWA, lives in stores.
- iOS app-icon set: 20, 29, 40, 58, 60, 76, 80, 87, 120, 152, 167, 180.
- Android adaptive icon: 108×108 foreground + 108×108 background SVG.

`@capacitor/assets` generates all of these from a single 1024×1024
master, so this is one designer task, not 17.

## Build pipeline scripts

`package.json` exposes:

- `build:web` (alias of `build`) — runs today.
- `build:ios` — stub. After Capacitor install: `pnpm build && cap sync ios && cap build ios`.
- `build:android` — stub. After Capacitor install: `pnpm build && cap sync android && cap build android`.

Stubs `exit 1` to make CI failures obvious until the real toolchain
is wired.

## Live-reload caveat

`capacitor.config.ts` sets `server.url` to the Vercel deployment. This
is the "ship instantly" superpower — but it means the native shell
**always** loads the current production build. If a release is broken,
every native installation is broken. Add a dual-channel option later:

- `production` shell → `courier-beta-seven.vercel.app`
- `staging` shell → preview deployment URL

For now, single channel is fine.
