## Report-only findings (prioritized)

### [P2] `orders/[id]/page.tsx` — STATUS_LABEL and PAYMENT_LABEL missing diacritics (operator-facing)

- **Where:** `apps/restaurant-admin/src/app/dashboard/orders/[id]/page.tsx` lines 11–27
- **Symptom:** Status labels shown on the order detail page use bare Romanian without diacritics: `In asteptare`, `Confirmata`, `Trimisa`, `Livrata`, `Anulata`; payment labels: `Neplatita`, `Platita`, `Rambursata`, `Esuata`.
- **Suggested fix:** Apply diacritics to each label. Not auto-fixed in this run because PR #4 already touches this file (`Marcheaza plata cash primita`); a new PR on top of #4 would have merge conflicts. Fix in next run after #4 is merged.

### [P2] ESLint not configured — `next lint` prompts interactively

- **Where:** Both `apps/restaurant-web` and `apps/restaurant-admin` have `"lint": "next lint"` in `package.json` but no `eslint.config.js` or `.eslintrc.json` in the app root.
- **Symptom:** Running `pnpm -r lint` causes `next lint` to enter an interactive wizard — CI/CD would hang or fail. Lint is effectively bypassed.
- **Suggested fix:** Add a minimal `eslint.config.mjs` (Next.js strict preset) to each app. Not auto-fixed because it requires creating a new file with `next/core-web-vitals` config — outside the 50-line cap and touches build infrastructure.

### [P2] `markCodOrderPaid` in `orders/actions.ts` lacks defensive `try/catch` for `payment_method` column

- **Where:** `apps/restaurant-admin/src/app/dashboard/orders/actions.ts` lines 103–109
- **Symptom:** Unlike `orders/page.tsx` and `orders/[id]/page.tsx` which both have defensive fallback selects for the `payment_method` column (migration 20260504_001), the `markCodOrderPaid` server action selects `id, payment_method, payment_status` without a fallback. Since the migration has shipped, this is currently safe, but the pattern is inconsistent.
- **Suggested fix:** Wrap the select in a try/catch that falls back to legacy columns. The function would need to assume non-COD if the column is absent. Risk is low now that migration is applied.
- **Why not auto-fixed:** The legacy fallback behaviour for `markCodOrderPaid` requires choosing a safe default for an unknown payment_method — wrong assumption could silently prevent a COD mark-paid. Needs human judgment.

### [P2] `integration_providers` table has no seed rows — adapter scaffolding unexercised

- **Where:** `supabase/migrations/20260501_001_integration_core.sql`, `apps/restaurant-admin/src/app/dashboard/settings/integrations/`
- **Symptom:** The `integration_providers` table exists with RLS, CRUD admin UI, and cron job — but no provider has ever been inserted for any tenant. The cron-based pull integration has never fired in production. Until an operator configures a provider, the entire integration-core scaffolding is dead code.
- **Suggested fix (P2):** Add a short onboarding prompt on the integrations settings page ("No integrations configured. Connect your first POS or courier system below.") so operators know the feature exists and how to enable it. Schema migration not needed.
