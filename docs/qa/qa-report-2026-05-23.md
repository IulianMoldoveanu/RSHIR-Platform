### [P1] Migration duplicate prefixes — 5 new groups (carry-forward + new since 2026-05-16)

- **Where:** `supabase/migrations/`
- **New groups this run:**
  - `20260605_003` — 3 files: `audit_log_chain_hardening`, `mv_refresh_pipeline`, `support_messages`
  - `20260605_004` — 3 files: `courier_order_secrets_table`, `status_page`, `support_replies`
  - `20260606_003` — 2 files: `health_check_pings_payload`, `payment_lifecycle`
  - `20260608_003` — 3 files: `marketing_drafts`, `menu_agent_proposals`, `whatsapp_business_skeleton`
  - `20260609_001` — 2 files: `pre_orders`, `voice_calls`
- **Total duplicate groups across repo (now 18):** see prior reports for earlier groups.
- **Symptom:** On a fresh Supabase restore, Supabase applies migrations in lexicographic order. When two files share the same `YYYYMMDD_NNN_` prefix, order of application is non-deterministic (filesystem sort). If one creates a table that another ALTERs, the clean-install will fail intermittently.
- **Suggested fix:** Rename the less-critical file in each pair/triple to the next available sequential slot. E.g. `20260605_004_mv_refresh_pipeline → 20260605_005_mv_refresh_pipeline`.
- **Why not auto-fixed:** `supabase/migrations/**` is read-only for the QA agent.
