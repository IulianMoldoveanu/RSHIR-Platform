### Fix 1 — `fix(referrals): repair partner attribution pipeline`
**Commit:** `b500e9b`
**Files:** `apps/restaurant-web/src/app/migrate-from-gloriafood/_components/lead-forms.tsx` · `apps/restaurant-web/src/app/migrate-from-gloriafood/page.tsx` · `apps/restaurant-admin/src/app/api/signup/route.ts`

Four silent failures that together made every `?ref=` partner referral signup fail silently:

1. **`ref` is a reserved React prop** — `<LeadForms ref={refCode} />` in `page.tsx:328` — React intercepts `ref` for DOM forwarding and never passes it as a prop. `refCode` inside `LeadForms` was always `''`. Renamed prop to `refCode` in the type + function signature.

2. **Wrong column in partner lookup** — `signup/route.ts:136` used `.eq('code', ref)` but `partners` table has no `code` column (see migration `20260507_003`). Partner referral links use `partner.id` (UUID), so the correct query is `.eq('id', ref)`.

3. **Non-existent `source` column in insert** — `signup/route.ts:145` wrote `source: 'self_serve'` into `partner_referrals`, which has no such column; every attribution insert errored silently.

4. **`ref` schema capped at 30 chars, UUIDs are 36** — `signup/route.ts:26` `.max(30)` rejected every UUID-based referral link with 400 before tenant creation. Changed to `.max(36)` (also removed `.toLowerCase()` since UUID case is irrelevant to lookup by id).

Flagged by Codex review on merged PR #98 (3 comments) and PR #97 (1 comment). PRs were merged so Codex comments are SKIP per §6.2, but bugs were present on main and fixed here.

Safety net: `pnpm -r typecheck` ✅ · `restaurant-web build` ✅ · `restaurant-admin build` ✅

- **Symptom:** `<img>` tags use `h-full w-full object-cover` in a fixed container but have no explicit `width`/`height` attributes. The browser technically has the container dimensions, so CLS impact is minimal, but the Next.js `no-img-element` rule and HTML spec both expect explicit dimensions.
