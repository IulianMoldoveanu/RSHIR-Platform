## Report-only findings (prioritised)

### [P2] `/rezervari` storefront page fully hardcoded in Romanian — no i18n

- **Where:** `apps/restaurant-web/src/app/(storefront)/rezervari/form.tsx`, `page.tsx`, `actions.ts`
- **Symptom:** All UI strings (field labels, success message, error messages, submit button, back-link, page title, metadata) are hardcoded Romanian. EN visitors see RO copy. No `t(locale, '...')` calls anywhere in the module.
- **Affected strings (non-exhaustive):** `'Rezervare trimisă'`, `'Trimite rezervarea'`, `'Se trimite…'`, `'Numele tău *'`, `'Telefon *'`, `'Persoane *'`, `'Data *'`, `'Ora *'`, `'Mențiuni (opțional)'`, plus all error messages in `actions.ts`.
- **Suggested fix:** Add a `rezervari` section to `dictionaries.ts` (RO + EN) and thread `getLocale()` → `t(locale, 'rezervari.*')` through the page. The `page.tsx` already uses `getLocale()` patterns (see `/checkout`, `/m/[slug]`). For `actions.ts`, return error keys (e.g. `'rezervari.err_invalid'`) and translate on the client.
- **Why not auto-fixed:** Touches 4 files (`dictionaries.ts`, `page.tsx`, `form.tsx`, `actions.ts`) with >50 lines total change; exceeds auto-fix cap.

### [P2] Duplicate migration prefix `20260506_001_*`

- **Where:** `supabase/migrations/20260506_001_copilot_daily_brief_schema.sql` and `supabase/migrations/20260506_001_orders_payment_status_index.sql`
- **Symptom:** Supabase applies migrations in lexicographic order by filename prefix. Two files sharing `20260506_001_` may apply non-deterministically, and one will be skipped if the Supabase migration runner tracks by prefix.
- **Suggested fix:** Rename `20260506_001_orders_payment_status_index.sql` → `20260506_002_orders_payment_status_index.sql`.
- **Why not auto-fixed:** `supabase/migrations/**` is read-only for this agent.
- **Note:** Previously reported in PR #45 (earlier run). Still unresolved.
