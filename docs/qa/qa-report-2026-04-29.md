## Report-only findings (prioritized)

### [P2] Duplicate migration name prefix `20260506_001_*`
- **Where:** `supabase/migrations/20260506_001_copilot_daily_brief_schema.sql` and `supabase/migrations/20260506_001_orders_payment_status_index.sql`
- **Symptom:** Two migration files share the prefix `20260506_001_`. Supabase applies migrations in lexical order by filename; with identical prefixes the order between these two files is deterministic (alphabetical: `copilot_daily_brief_schema` before `orders_payment_status_index`) but fragile — any future tooling that expects a strict monotonic sequence will report an integrity error. The Supabase dashboard may also flag this in the migration history.
- **Suggested fix:** Rename one of them to `20260506_002_orders_payment_status_index.sql`. Requires coordination with anyone running migrations locally (they'll need to rename + re-apply in their local Supabase instance).
- **Why not auto-fixed:** Never write to `supabase/migrations/` (hard guardrail). Requires human rename.

### [P3] Integration adapter registry has only `mock` registered
- **Where:** `packages/integration-core/src/adapters/registry.ts`
- **Symptom:** The `REGISTRY` object only contains `mock`. Any tenant in the database configured with a real provider key (`iiko`, `smartcash`, etc.) will throw `"No integration adapter registered for provider '…'"` at runtime. The PR #43 batch merge included the integration-core scaffold but no real adapters. If `integration_providers` rows with non-mock keys exist in any environment, those tenants will see errors on every webhook.
- **Suggested fix:** Either ensure no non-mock rows exist in dev/prod (seed guard), or implement the first real adapter. Low urgency if no production tenants use integrations yet.
- **Why not auto-fixed:** Implementing an adapter is a feature, not a bug fix.
